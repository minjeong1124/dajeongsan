## 1. 프로젝트 개요 (Overview)

**프로젝트명:** 다정산 (Dajeongsan)

**배경 및 문제 정의:**

- 친구·지인 모임에서 정산 요청자는 정확한 금액(백원/천원 단위)을 그대로 요구할 때 인색해 보일까봐 부담을 느낀다.
- 이런 부담은 단순히 "돈 얘기가 어색하다"는 수준을 넘어, 정확한 금액의 정산을 요청하거나, 정산 처리를 독촉하는 행위 자체가 관계 안에서 "너무 정확히 따지는 사람"이라는 인상을 남길까 하는 걱정으로 이어진다
- 친한 관계에서는 돈을 정확히 나누는 행동이 관계보다 손익을 앞세우는 태도로 해석될 수 있기 때문이다. 친구나 지인 관계에서는 약간의 손해를 감수하는 태도가 호의나 친밀함의 신호로 받아들여질 수 있다.
- 특히 상대가 여러 번 밥을 샀거나, 서로 가까운 관계라고 느낄수록 사용자는 “이 정도까지 받아도 되나?”를 고민한다. 그래서 실제 정산 금액보다 관계 안에서의 인상이 더 중요하게 작동한다.
- 기존 대안(카카오페이 정산하기 등)은 계산의 정확성과 송금 편의는 해결했지만, 그 정확함이 한국 정서 상 인간관계 안에서 유발하는 사회적 부담(쪼잔해 보임, 독촉의 어색함)은 다루지 않는다.
- 또한 계산 결과를 정확하게 보여주기 때문에, 사용자가 금액을 낮추어 정산 기능 외 별도의 메세지 발송을 병행하는 행동은 단순 실수가 아니라 의식적인 조정이 되고, "정확히 할 수 있는데 안 하는 것"이 되어 오히려 선택의 문제(내가 여유를 보일지 말지)로 부담이 전가 될 수 있다.
- 기술이 정확함을 "기본값"으로 만들어 계산의 정확성은 높였지만, 정확한 금액이 친밀한 관계 안에서는 부담으로 읽힐 수 있다는 점은 충분히 반영하지 못했다. 그래서 사용자는 의식적으로 기술을 거스르는 행동(앱 밖에서 직접 금액을 수정하거나 말투를 조정)을 직접 해야 하는 추가 노동이 생겼다.

**목표:**

- 정산 요청을 "정확한 금액 독촉"이 아니라 "자연스러운 배려가 담긴 요청"으로 전환한다.
- 정산 요청자는 정확한 금액을 요구할 때 쪼잔해 보일 수 있다는 부담을 느끼고, 요청을 미룰수록 상대가 잊거나 응답하지 않아 다시 말해야 하는 상황을 겪을 수 있다. 이에 정산 처리(확인) 시점에 따라 절삭 금액과 배려 문구를 자동으로 제안하고, 관계별 누적 절삭 금액을 함께 보여주면, 사용자는 빠른 정산 요청을 독촉이 아니라 상대의 빠른 처리에 배려로 보답하는 정리 방식으로 전달할 수 있으며, 배려가 일방적 손해가 아니라 상호적으로 오가는 경험이라고 느낄 수 있다.
- 즉, 정산 요청 시 상대의 정산 완료 시점에 따라 절삭 혜택을 제공함으로써, 사용자의 정산 요청은 '빨리 돈을 보내달라'는 독촉이 아니라 '빨리 처리해주면 내가 조금 덜 받을게'라는 배려형 요청으로 전환될 수 있다는 것이 핵심 목표다.
- 이를 통해 다음 다섯 가지를 함께 해소하는 것을 목표로 한다
    1. 정확한 금액 요청이 쪼잔해 보일 수 있다는 부담 완화
    2. 지불자의 정산 처리를 독촉하는 민망함 완화
    3. 지불자의 정산 처리를 미루다 재요청 상황으로 이어지는 것을 사전에 예방
    4. 나만 계속 깎아주는 것 같은 일방적 손해감 완화 (상호성 확인)
    5. 절삭 기준이 매번 달라 생기는 형평성 문제 해소 (시간·설정 기준의 일관된 적용)

## 2. 핵심 기능 요구사항 (Requirements)

**사용자 플로우:**

1. 정산 요청 생성 (총액+인원 N분의1 또는 참여자별 개별 금액 직접입력 선택 / 절삭 토글 카드 노출—미리보기 문구 포함, default off, 켜면 전역설정으로 유지)
2. ➔ 참여자별 개별 링크 발급 및 공유
3. ➔ 지불자, 개별 링크 접속 (신규 브라우저인 경우 매칭용 계좌 뒷자리 4자리 입력)
4. ➔ 절삭 토글 상태에 따라 열람시각 기준 절삭 판정+배려문구 또는 정확 금액 노출
5. ➔ 요청자, 대시보드 재방문 시 참여자별 열람상태·최종금액·관계별 누적 절삭 기록 확인
6. ➔ 지불자가 다음 모임에서 요청자가 되어 정산 요청 생성 시, 참여자들로부터 받아온 누적 절삭 금액을 노출하며 절삭 토글 사용 유도
7. ➔ 반복

**데이터 처리:**

- 저장소: Supabase
- 저장:
    - 요청 생성 시
        - 요청 ID
        - 요청자 식별자(익명토큰 또는 계좌 뒷자리)
        - 총액 또는 참여자별 개별금액
        - 참여자 이름
        - 절삭 토글 여부
        - 생성 시각
    - 지불자 최초 접속 시
        - 열람 시각
        - 판정된 절삭 등급, 확정 금액을 해당 참여자 레코드에 업데이트
        - 신규 브라우저(미인식 토큰)인 경우 → 매칭용 계좌 뒷자리 4자리 추가 저장
- 조회:
    - 지불자 링크 접속 시
        - 절삭 토글이 켜져 있으면 생성 시각 대비 경과 시간으로 절삭 등급(999원/99원/없음) 판정 후 배려 문구와 함께 노출, 꺼져 있으면 정확한 금액만 노출
    - 요청자 대시보드 접속 시
        - 참여자별 열람 여부·확정 금액 노출 + 동일 식별자(계좌 뒷자리 매칭) 기준 과거 절삭 누적액 집계 노출
    - 사용자가 과거에 지불자였던 이력이 있는 상태로 새 요청을 생성하려 할 때
        - 그동안 참여자들에게서 받은 누적 절삭액을 조회해 요청 생성 화면에 노출

**필수 기능 목록:**

- 정산 요청 생성 — 총액+인원 기준 N분의1 균등분배 또는 참여자별 개별 금액 직접입력 중 선택 가능
- 절삭 토글 카드 — 요청 생성 화면에 기본 off 상태로 노출되며(사용자 관성을 깨지 않기 위함), 켜면 배려 문구 미리보기 예시가 함께 표시됨. 한 번 켜면 전역 설정으로 계속 유지됨
- 참여자별 개별 링크 발급 (1:N 구조) — 지불자별로 각자 다른 링크를 받아 개인별 열람 시각을 독립적으로 추적
- 시간 기반 절삭 로직 — 절삭 토글이 켜진 경우, 링크 최초 열람 시각 기준 1시간 이내 999원 미만, 5시간 이내 99원 미만, 이후 절삭 없음
- 배려형 문구 자동 노출 — 절삭 적용 시 지불자 화면에 절삭 금액과 자연스러운 배려 문구 함께 표시
- 공유용 텍스트/링크 생성 — 참여자별 링크를 카카오톡에 붙여넣기 좋은 형태로 자동 생성
- 요청자용 상태 대시보드 — 참여자별 열람 여부, 확정 금액, 관계별 누적 절삭 기록 확인
- 사용자 식별 및 매칭 — 평소에는 브라우저 익명 토큰으로 식별하고, 신규 브라우저 접속 시에는 매칭용으로 계좌 뒷자리 4자리를 입력받아 과거 이력과 연결
- 재사용 유도 — 과거에 지불자였던 사용자가 새 요청을 생성할 때, 그동안 참여자들로부터 받은 누적 절삭액을 보여줘 절삭 토글 사용을 자연스럽게 유도
