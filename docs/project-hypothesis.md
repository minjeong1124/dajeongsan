## 핵심 가설

정산 요청 시 상대의 정산 완료 시점에 따라 절삭 혜택을 제공하면,
사용자의 정산 요청은 ‘빨리 돈을 보내달라’는 독촉이 아니라
‘빨리 처리해주면 내가 조금 덜 받을게’라는 배려형 요청으로 전환될 수 있다.

정산 요청자는 정확한 금액을 요구할 때 쪼잔해 보일 수 있다는 부담을 느끼고,
요청을 미룰수록 상대가 잊거나 응답하지 않아 다시 말해야 하는 상황을 겪을 수 있다.

따라서 정산 처리 시점에 따라 절삭 금액과 배려 문구를 자동 제안하고, 관계별 누적 절삭 금액을 함께 보여주면,
사용자는 빠른 정산 요청을 독촉이 아니라 상대의 빠른 처리에 배려로 보답하는 정리 방식으로 전달할 수 있으며,
배려가 일방적 손해가 아니라 상호적으로 오가는 경험이라고 느낄 수 있을 것이다.

## 가설이 해결하려는 문제

| 문제 | 해결 방식 |
| --- | --- |
| 정확한 금액 요청이 쪼잔해 보일 수 있음 | 절삭 금액 + 배려 문구로 요청자의 이미지를 완충 |
| 정산 처리를 독촉하기 민망함 | 빠른 처리일수록 더 큰 배려를 붙여 “정산 지시 또는 독촉”이 아니라<br>“배려 + 빠른 정산 처리에 대한 보답”으로 프레이밍 |
| 정산 처리를 미루다 재요청 상황으로 이어짐 | 금액 할인으로 정산 요청 직후 처리를 유도해 에스컬레이션 이전 단계에서 문제를 예방 |
| 나만 계속 깎아주는 것 같은 손해감 | 관계별 누적 절삭 금액을 보여줘 상호성을 확인 |
| 절삭 기준이 매번 달라 생기는 형평성 문제 | 시간 기준과 설정 기준으로 일관되게 적용 |

## 핵심 기능

| 핵심 기능 | 내용 | 해결하는 문제 |
| --- | --- | --- |
| 1. 시간 기반 정산 금액 절삭 | 정산 요청 후 1시간 이내 처리 시 1,000원 미만 절삭, 5시간 이내 요청 시 100원 미만 절삭, 이후에는 정확 금액 요청 | 빠른 정산에 대한 요청을 “돈을 빨리 달라는 행동”이 아니라 상대에게 배려를 담아 정리하는 행동으로 전환 |
| 2. 배려형 정산 문구 자동 생성 | 절삭된 금액과 함께 “끝자리는 내가 덜었어”, “편하게 이 금액만 보내줘”처럼 요청자의 배려가 느껴지는 문구 자동 삽입 | 정확한 금액을 요구할 때 생기는 쪼잔함/계산적 이미지 부담 완화 |
| 3. 관계별 누적 절삭 금액 노출 | 과거에 서로 주고받은 절삭 금액을 요청자에게만 보여줌. 예: “지난번에 민지님이 700원 덜어줬어요” | 절삭이 일방적 손해처럼 느껴지지 않도록 하고, 상호 배려와 형평성 유지 |

## 기대 효과

- 사용자 관점 : 정산 요청 작성에 드는 심리적 고민(할인 여부 판단, 문구 작성)이 사라짐 — 시스템이 항상 일관되게 처리
- 사용자 관점 : "인색해 보일까"라는 걱정과, 나만 손해 보는 것 같은 불안 및 형평성 문제를 동시에 해결
- 사용자 관점 : 정산이 딱딱한 돈 계산이 아니라 서로의 배려가 오가는 경험으로 전환
- 사용자 관점 : 누적 차이가 클 경우 정확 금액 요청을 선택할 명분 확보
- 비즈니스 관점 : 계산 정확성 중심의 정산 서비스와 다른 관계 배려형 포지셔닝 가능
- 비즈니스 관점 : 정산 요청 발송률, 빠른 요청률, 완료율 개선 가능성

## 고려한 리스크와 대응 방법

| 리스크 | 설명 | 대응 |
| --- | --- | --- |
| 누적 손실 부담 | 자주 정산하는 사용자는 손실이 쌓일 수 있음 | 소액 절삭으로 제한.<br>백원 미만 절삭 시 3개월 11회 이상 정산해도 최대 약 1,089원 수준<br>+ 상대와의 절삭 기록을 과거에 절삭 정산을 받은 사용자가 같은 상대에게 정산 요청할 때 표시 -> 상대 절삭 유도하여 일방적 손해감 형평성 우려 완화 |
| 사용자가 정확한 금액을 원할 수도 있음 | 모든 사용자가 금액 절삭을 선호하지는 않음 | “정확한 금액으로 요청” 옵션 유지 |
| 소액 정산에서 오히려 손해가 크게 느껴질 수 있음 | 5,000원 이하 정산에서 99원 절삭도<br>상대적으로 크게 느껴질 수 있음 | 5,000원 미만은 절삭 생략<br>또는 별도 기준 적용 검토 |
| 진짜 배려 신호가 사라질 위험 | 자동화되면 사용자의 호의가 덜 진정성 있게 보일 수 있음 | 정확 금액 요청 옵션 유지, 문구 톤을<br>과하지 않게 설계 |

## 예시 방향

| 요소 | 설계 방향 |
| --- | --- |
| 기능명 | 빠른 정산 배려 모드 |
| 1시간 이내 정산 요청 | 1,000원 미만 절삭 (최대 999원) |
| 5시간 이내 정산 요청 | 100원 미만 절삭 (최대 99원) |
| 5시간 이후 정산 요청 | 절삭 없음 |
| 문구 | 요청자의 배려처럼 보이는 문구 자동 삽입 |
| 정확 금액 옵션 | “정확한 금액으로 요청” 유지 |
| 설정 | 요청마다 선택하지 않고, 설정에서 1회 on/off |
| 절삭 기록 | 관계별/상대별 누적 절삭 금액 표시 |
| 절삭 기록 노출 시점 | 과거에 절삭 정산을 받은 사용자가 같은 상대에게 정산 요청할 때 표시 |
| 절삭 기록 노출 목적 | 절삭을 유도하되, 일방적 손해감과 형평성 우려 완화 |

## 절삭 기록 예시

| 상황 | 노출 정보 | 의도 |
| --- | --- | --- |
| 내가 과거에 A에게 절삭해준 적 있음 | “A님에게 지금까지 1,200원 배려했어요” | 내 배려 기록 확인 |
| A가 과거에 나에게 절삭해준 적 있음 | “A님이 지난 정산에서 800원 배려해줬어요” | 상호성 인식 |
| 상대가 더 많이 절삭함 | “지난번에 받은 배려를 이번에 이어갈 수 있어요” | 절삭 유도 |

| 리스크 | 설명 | 대응 |
| --- | --- | --- |
| 관계를 점수화하는 느낌 | “누가 더 깎아줬는지” 비교가 노골적이면 불쾌할 수 있음 | ‘정산 점수’가 아니라 ‘배려 히스토리’로 부드럽게 표현 |
| 상대 감시처럼 보임 | 상대가 얼마나 덜어줬는지 추적하는 느낌 | 개인에게만 노출, 상대에게 공개하지 않음 |
| 손익 계산처럼 보임 | “내가 더 손해 봤다”를 강조하면 관계 부담 증가 | 정확 금액 요청을 자연스럽게 제안하는 정도로 사용 |
| 절삭 강요 | 상대가 해줬으니 나도 해야 한다는 압박 | “선택 가능” 톤 유지 |
